Formålet med denne guide

Denne guide er en intern reference til at sætte et komplet, moderne Business Central udviklingsmiljø op fra bunden — med Claude Code og Cursor som primære værktøjer og de nye AL Development Tools (altool) som rygrad.

Guiden er målrettet en situation hvor maskinen er reset eller en ny BC-udvikler skal i gang: hvad skal installeres, i hvilken rækkefølge, hvor placeres konfigurationsfilerne, og hvordan kobler det hele sammen. Hver kodeblok kan kopieres direkte — udfyld dine egne værdier i Variabler-sektionen, så indsættes de automatisk i alle eksempler.

Målet er en arbejdsstation hvor Claude Code har semantisk AL-forståelse, kan tale direkte med BC, kan committe til GitHub, og hvor Cursor er din editor med fuld AL-IntelliSense — alt sammen med konsistente projektregler og minimalt token-forbrug.

Reference
LSP
AL LSP Server
Sprogintelligenens ›

LSP står for Language Server Protocol — en standard som Microsoft skabte, hvor en editor kommunikerer med en separat "sprogserver" for at få funktioner som go-to-definition, find-references, rename, completions og fejlmarkering. Selve sprogforståelsen bor i serveren, ikke i editoren.

AL LSP Server er motoren bag den intelligens der driver IntelliSense i VS Code's AL-extension. Den taler LSP over stdio (JSON-RPC) og giver fuld navigation: go-to-definition, find-references på tværs af projekter (resolver internalsVisibleTo/propagateDependencies), rename, completions og type hierarchy. Den kører som editorens sprog-host og er beregnet til editorer og LSP-kompatible agenter.

Claude Code arbejder over MCP. Via søster-serveren altool launchmcpserver (begrebet AL MCP nedenfor) får Claude Code opgave-værktøjer — al_build, al_compile, al_publish, al_symbolsearch og al_getdiagnostics. Det er stærkere end grep til symbol-søgning og fejl-tjek. Den fulde find-references/rename bor i LSP-serveren og kan kobles på Claude Code via en LSP-bro.

Relateret: launchmcpserver registreres i .claude.json til Claude Code — se sektion 2 og 4.
MCP
MCP Server (Model Context Protocol)
Bro til systemer ›

MCP er en åben protokol som Anthropic publicerede i slutningen af 2024, der lader AI-assistenter (som Claude) tale med eksterne systemer via en standardiseret JSON-RPC interface over stdio eller HTTP. Tænk på det som "USB-porten for AI-værktøjer".

En MCP server eksponerer værktøjer og data som Claude kan kalde. F.eks.:

  • AL LSP MCP → AL kode-intelligenens
  • BC MCP Server → direkte adgang til BC's API og data (BC 27.1+, public preview)
  • GitHub MCP → repo management, PR reviews, AL-Go pipelines
  • Filesystem MCP → lokale filer uden for projektmappen

Du registrerer hvilke MCP-servers Claude må bruge i ét af tre scopes: user (~/.claude.json, alle projekter), project (.mcp.json i projektroden, kan checkes ind) eller local (privat pr. projekt). Herefter kan Claude kalde dem automatisk når en opgave kræver det — se Trin 4 for den konkrete opdeling.

Relateret: Se sektion 3 for prioriteret liste af MCP servers og sektion 4 for konkret JSON-konfiguration.
SKL
Skill
Pakket ekspertise ›

En Skill er en mappe med en SKILL.md fil plus eventuelle scripts og reference-data. Mappen pakker en specifik ekspertise — f.eks. "AL-kodestil", "vendor-matching" eller "OIOUBL schematron-regler" — så Claude kan loade præcis den viden når opgaven kræver det.

Det smarte: Claude loader kun de filer der er relevante for den aktuelle opgave. Du kan have et dusin reference-filer i en skill, men hvis opgaven kun kræver vendor-schemaet, læser Claude bare den ene fil. Det holder token-forbruget nede.

SKILL.md har frontmatter med name og description — det er beskrivelsen Claude bruger til at vurdere om skill'en er relevant for en given prompt.

Relateret: Skills bor i ~/.claude/skills/ — se sektion 5 for struktur og eksempel.
SUB
Subagent
Isoleret kontekst ›

En Subagent er en Claude-instans der kører i sin egen isolerede kontekst. Den modtager en specifik opgave fra hovedagenten, løser den, og returnerer kun resultatet — ikke al sin tankegang og mellemregninger.

Det er værdifuldt fordi det holder hovedkonteksten ren. Hvis hovedagenten f.eks. arbejder på AL-kode og du beder den parse 50 OCR-fakturaer, ville al den OCR-data forurene konteksten. En subagent kan klare OCR-parsingen og kun rapportere "50 fakturaer behandlet, 47 matched, 3 kræver manuel review" tilbage.

I praksis: i ~/.claude.json kan du sætte CLAUDE_CODE_SUBAGENT_MODEL til haiku — så kører subagenter på en billig, hurtig model mens hovedagenten bliver på Opus/Sonnet. Det passer til subagenternes grovarbejde (søg, parse, opsummér) og holder token-forbruget nede. Bemærk: denne env-variabel har højeste prioritet og tvinger alle subagenter over på den model — vil du have finkornet styring (fx en review-agent på Opus), så lad den være usat og sæt i stedet model: pr. agent.

Relateret: Bruges automatisk af Claude Code når den opdager parallelle eller isolerbare opgaver.
.md
CLAUDE.md & .cursorrules
Projektregler ›

CLAUDE.md er en markdown-fil Claude Code automatisk læser ved start af en session i et projekt. Den indeholder projektets "husregler": kodestil, navngivningskonventioner, kendte afhængigheder, foretrukne mønstre. Den bor i {PROJEKT}/.claude/CLAUDE.md.

.cursorrules er det tilsvarende for Cursor's Composer og Chat — samme idé, anden fil. Den bor i projektets rod.

De to filer kan have meget af det samme indhold, men det er smartest at holde dem som kilden til sandheden for hver editor. Skal du sikre konsistens på tværs, kan du symlinke dem eller have en master RULES.md begge læser fra.

Relateret: Se eksempler i sektion 4.
BCA
BC Agent & AI Development Toolkit
BC 27.4+ ›

I februar 2026 åbnede Microsoft en ny vej til at bygge AI-agenter direkte i Business Central: AI Development Toolkit. Du kan designe en agent i BC's sandbox-UI uden at skrive AL — teste prompts mod live data, validere adfærd, og når du er klar, eksportere til en AL-implementation.

Det er en anden ting end Claude Code's subagents — BC-agenten kører inde i BC og bruger Microsoft's egne AI-modeller (GPT-5.3-chat fra maj 2026). Den er relevant for end-user-features som "Payables Agent" eller en custom vendor-/faktura-agent.

Claude Code er din udviklerassistant uden for BC — BC Agent er en feature du bygger ind i BC til slutbrugerne.

Relateret: Se sektion 5, walkthrough-videoer på aka.ms/BCDesignAgents.
PC
Prompt Caching
Token-besparelse ›

Når du sender en stor blok statisk kontekst til Claude (f.eks. en vendor-liste på 5000 linjer, et AL-schema, eller et OCR-faktura-JSON-spec), kan du markere den med cache_control: ephemeral. Anthropic gemmer da en intern repræsentation af blokken i 5 minutter.

Næste gang du sender en prompt med samme cache-markerede blok, bruger Claude den cachede version — det koster ~10% af normal pris for den del af konteksten. På gentagne kald med store statiske blokke kan du spare op til 82%.

Praktisk eksempel: din vendor-agent loader 5000 kreditorer hver gang den behandler en faktura. Cache den — så koster faktura nr. 2 til 100 næsten ingenting på vendor-list-konteksten.

Relateret: Se sektion 8 for konkrete optimerings-mønstre.
HK
Hooks
Automatisering ›

Hooks er shell-kommandoer der kører automatisk på bestemte events i Claude Code — før en tool-call, efter en fil-redigering, ved session-start. Det giver dig kontrol over kvalitet og sikkerhed uden at stole på at Claude husker det selv.

Eksempler: auto-format AL-filer efter hver redigering, bloker farlige kommandoer som Uninstall-NAVApp på produktion, log token-spend efter hver session, eller cache vendor-liste ved session-start.

Hooks konfigureres i ~/.claude/settings.json under en hooks nøgle med event-typer som PreToolUse, PostToolUse, SessionStart og Stop.

Relateret: Se sektion 8 for konfiguration og BC-specifikke eksempler.
OAU
BC OAuth & Entra App Registration
Authentication ›

BC's API'er (og dermed BC MCP Server) bruger OAuth 2.0 mod Microsoft Entra — samme model som AL-Go for GitHub og BC Cloud API'er. Det betyder en engangs-opsetning men nogle gotchas omkring token-refresh.

De to flows du vil møde:

  • Client Credentials — server-to-server, refresh via secret. Anbefales til BCMCPProxy.
  • Authorization Code — interaktiv brugerlogin, refresh token holder 90 dage.

Access tokens udløber efter 1 time — BCMCPProxy håndterer refresh automatisk hvis konfigureret korrekt. Når du ser 401 efter 1 time er det altid en refresh-fejl.

Relateret: Sektion 9 har step-by-step setup og fejlfindings-tabel for almindelige AADSTS-fejlkoder.
CI
Claude i AL-Go pipelines
CI/CD ›

claude --print (eller -p) gør at Claude kan køre non-interaktivt i CI/CD. Du sender en prompt, Claude udfører opgaven, skriver til stdout, og afslutter — ingen interaktivitet, ingen permission-prompts.

Det åbner for AI-augmenterede AL-Go workflows: automatiske code reviews på PR'er, oversaettelse af nye xlf-labels, breaking-change analyse når BC major-versioner skifter, og auto-genererede release notes baseret på commits.

Omkostninger styres ved model-routing: Haiku til oversaettelse, Sonnet til reviews, Opus kun til kritisk analyse. Et typisk PR-review ligger på 2-15 cents.

Relateret: Sektion 10 har tre færdige workflow-eksempler du kan kopiere direkte.

Forudsætninger

Engangs · maskine
⚠
Installer i denne rækkefølge
Node.js skal installeres før Claude Code, da Claude Code er en npm-pakke. Verificer node --version viser 20+ inden du fortsætter.

A · Node.js 20+ via nvm-windows

PowerShell — nvm-windows
# Download nvm-windows: https://github.com/coreybutler/nvm-windows/releases
# Installer nvm-setup.exe, luk og genåbn PowerShell:
nvm install 20
nvm use 20
node --version    # skal vise v20.x.x

B · Claude Code

PowerShell
npm install -g @anthropic-ai/claude-code
# Kør én gang for at acceptere permissions
claude --dangerously-skip-permissions

C · AL extension → find altool.exe stien

PowerShell — find sti og tilføj til PATH
# Tjek begge mapper (Cursor og VS Code deler IKKE altid extensions)
# Din installerede version er fx: ms-dynamics-smb.al-18.0.2293710
Get-ChildItem "$env:USERPROFILE\.vscode\extensions" -Filter "ms-dynamics-smb.al-*" | Select-Object Name
Get-ChildItem "$env:USERPROFILE\.cursor\extensions" -Filter "ms-dynamics-smb.al-*" | Select-Object Name

# Find altool.exe automatisk — AL-kommandolinjeværktøjet i extensionen
$alExe = Get-ChildItem "$env:USERPROFILE\.vscode\extensions\ms-dynamics-smb.al-*" -Recurse -Filter altool.exe -ErrorAction SilentlyContinue | Select-Object -First 1
$alPath = Split-Path $alExe.FullName
$alPath    # bekræft stien (...\ms-dynamics-smb.al-18.0.2293710\bin\win32)

# Tilføj til User PATH — læs User-scope (IKKE $env:PATH, der også rummer system-stier)
$userPath = [Environment]::GetEnvironmentVariable("PATH", "User")
[Environment]::SetEnvironmentVariable("PATH", $userPath + ";$alPath", "User")
$env:PATH += ";$alPath"    # opdatér også den nuværende session — undgå genstart

# Verificer at altool svarer:
altool launchmcpserver --help

D · Git multi-account

PowerShell
git config --global credential.useHttpPath true
git config --global credential.helper manager
# Kør pr. repo efter clone:
cd "C:\AL\MyProject"
git config user.email "peter@kretzmann.dk"
git config user.name "Peter Kretzmann"
TRIN 02

AL MCP Server — altool launchmcpserver

EngangsPr. projekt
✦
AL Development Tools (AL-extension 17+)
AL-extensionen leverer AL-værktøjer på kommandolinjen via altool.exe (i bin\win32) — uden VS Code. Kommandoen altool launchmcpserver starter en MCP-server der eksponerer AL-værktøjer (al_build, al_symbolsearch, al_getdiagnostics, al_publish m.fl.) til AI-agenter — Claude Code er nævnt eksplicit som klient. Det er denne du registrerer i projektets .mcp.json (se Trin 4).
💡
altool — AL-kommandolinjeværktøjet
Kommandolinjeværktøjet hedder altool og ligger i AL-extensionens bin\win32. Installerer du NuGet-pakken AL Development Tools separat, kan du kalde det med det korte alias al — det er samme værktøj. Microsofts egne eksempler bruger ofte al.

Hvad du gør i dette trin

  1. Bekræft at altool kan køres (PATH blev sat i Trin 1C).
  2. Test at MCP-serveren svarer — se Test manuelt nedenfor.
  3. Kopiér al-mcp-entryen og hav den klar. Den indsættes i projektets .mcp.json i Trin 4 sammen med bc-mcp.

Test manuelt

PowerShell — verificer at altool og serverne svarer
altool --help                   # alle kommandoer
altool launchmcpserver --help   # MCP-serveren og dens options

# Start MCP-serveren manuelt (Ctrl+C for at stoppe — den venter på JSON-RPC på stdin):
altool launchmcpserver --transport stdio

MCP-konfiguration — al-mcp entryen

Denne al-mcp-entry hører til projektets .mcp.json (project scope — samlet med bc-mcp i Trin 4). Vælg én af de to former:

JSON — A: minimal (agenten loader projektet undervejs)
{
  "al-mcp": {
    "command": "altool",
    "args": ["launchmcpserver", "--transport", "stdio"]
  }
}
JSON — B: loader projektet og symbolerne med det samme
{
  "al-mcp": {
    "command": "altool",
    "args": [
      "launchmcpserver", "--transport", "stdio",
      "C:\\AL\\MyProject",
      "--packagecachepath", "C:\\AL\\MyProject\\.alpackages"
    ]
  }
}
💡
A eller B?
Begge virker. B peger serveren direkte på dit projekt og din symbol-cache (.alpackages), så værktøjer som al_symbolsearch og al_compile kan svare fra start. A er kortere — her tilføjer agenten projektet undervejs med al_addproject. Begge stier og --packagecachepath er valgfri ifølge altool launchmcpserver --help. (Windows-stier i JSON skrives med dobbelt backslash — det gør variablen automatisk.)
💡
MCP og LSP — hvad hver giver
MCP-serveren (launchmcpserver) er den du registrerer i projektets .mcp.json. Den giver Claude Code opgave-værktøjer: al_build, al_compile, al_publish, al_downloadsymbols, al_symbolsearch og al_getdiagnostics.
LSP-serveren er motoren bag kode-navigation i VS Code/Cursor: go-to-definition, find-references på tværs af projekter (resolver internalsVisibleTo/propagateDependencies), rename, completions og type hierarchy. Den kører som editorens sprog-host (Microsoft.Dynamics.Nav.EditorServices.Host.exe). Enkelte AL-builds eksponerer den også standalone som launchlspserver, der kan kobles på Claude Code via en LSP-bro.

Hvad er BcMCPProxy og hvorfor skal jeg bruge det?
BC understøtter MCP, men kun mod Copilot Studio — ikke mod Claude Code. BcMCPProxy er et lille lokalt Node.js-program der løser det: det kører på din maskine, modtager MCP-kald fra Claude Code, og videresender dem til BC's API via OAuth.

Din Entra App Registration er credentials (Client ID + Secret) som BcMCPProxy bruger til at må tale med BC. De to hører sammen: app registration = adgangskort, BcMCPProxy = døren der bruger kortet.

Docs: configure-mcp-server · Kildekode: BCTech/BcMCPProxy

💡
To scopes: global (user) og projekt
Claude Code har tre MCP-scopes; her bruger vi to. User gemmes i ~/.claude.json og loader i alle projekter — her lægger vi github (selve tokenet kommer fra en OS-miljøvariabel, så det aldrig står i en fil). Project gemmes i .mcp.json i projektroden, loader kun det projekt og kan checkes ind — her lægger vi al-mcp og bc-mcp (ingen hemmeligheder; BCMCPProxy's secret bor i dens egen .env, se Trin 9). User ↔ Engangs, project ↔ Pr. projekt.

~\.claude.json — global (user scope)

%USERPROFILE%\.claude.json
{
  "env": {
    // Subagenter (søg, parse, opsummering) på billig+hurtig model — hovedagenten forbliver på Opus/Sonnet
    "CLAUDE_CODE_SUBAGENT_MODEL": "haiku",
    "MAX_THINKING_TOKENS": "8000",
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  },
  // User scope — loader i alle projekter
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"]
      // tokenet arves fra OS-miljøvariablen GITHUB_PERSONAL_ACCESS_TOKEN (sættes nedenfor) — står aldrig i en fil
    }
  }
}
🔒
GitHub-token som miljøvariabel — ikke i config
GitHub-MCP-serveren læser GITHUB_PERSONAL_ACCESS_TOKEN fra sit process-miljø og arver din bruger-miljøvariabel. Sæt den én gang, så ligger tokenet aldrig i en config-fil og er nemt at rotere.
PowerShell — sæt GitHub-token som User-miljøvariabel
[Environment]::SetEnvironmentVariable("GITHUB_PERSONAL_ACCESS_TOKEN", "ghp_xxxxxxxxxxxx", "User")
# Genstart terminal + Claude Code — User-scope miljøvariabler slår kun igennem i nye processer

# Test i et NYT vindue at den er sat (viser tokenet):
$env:GITHUB_PERSONAL_ACCESS_TOKEN

Portabel variant — ${CLAUDE_PROJECT_DIR}
Skal samme .mcp.json virke i ethvert AL-projekt (og checkes ind uden fast sti), kan al-mcp bruge Claude Codes egen projekt-variabel i stedet for en hardkodet sti:
.mcp.json — al-mcp args (portabel)
"args": ["launchmcpserver", "--transport", "stdio",
         "${CLAUDE_PROJECT_DIR}",
         "--packagecachepath", "${CLAUDE_PROJECT_DIR}/.alpackages"]
Claude Code ekspanderer ${CLAUDE_PROJECT_DIR} til projektmappen. Skråstreger holder JSON'en gyldig uden escaping.

~\.claude\settings.json

%USERPROFILE%\.claude\settings.json
{
  "model": "claude-sonnet-4-6",
  "maxTokens": 8192,
  // ANTHROPIC_API_KEY læses automatisk fra OS-miljøvariablen (sættes nedenfor) — står aldrig i en fil
  "tokenBudget": { "warnAt": 50000, "compactAt": 80000 }
}
🔒
API-nøgle som miljøvariabel — ikke i config
Claude Code læser ANTHROPIC_API_KEY automatisk fra OS-miljøet, så nøglen behøver ikke stå i settings.json (hvor den ellers ville ligge i klartekst). TRIN 08

Hooks — automatiser hvad der sker omkring Claude Code

Engangs · global
💡
Hvad er hooks?
Hooks er shell-kommandoer Claude Code kører automatisk på bestemte tidspunkter — før en tool-call, efter en fil-redigering, ved session-start. Det giver dig kontrol over kvalitet og sikkerhed uden at stole på at Claude husker det selv. Tænk på dem som Git hooks, men for AI-agenter.

Hook events — hvornår de fyrer

EventNår det fyrerTypisk brug
SessionStartVed opstart af Claude Code i et projektLoad vendor-liste, tjek branch, varm cache
UserPromptSubmitFør din prompt sendes til modellenTilføj ekstra kontekst, log prompts
PreToolUseFør Claude kører et værktøjBloker farlige kommandoer, kræv bekræftelse
PostToolUseEfter Claude har kørt et værktøjAuto-format, lint, validate
StopNår Claude afslutter et svarSend notifikation, log spend
SubagentStopNår en subagent afslutterLog subagent-resultater

Konfigurer hooks i settings.json

%USERPROFILE%\.claude\settings.json — udvidet med hooks
{
  "model": "claude-sonnet-4-6",
  "hooks": {
    // Auto-format AL filer efter Claude har redigeret dem
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": "powershell -Command \"if ($env:CLAUDE_FILE_PATH -match '\\.al$') { al format $env:CLAUDE_FILE_PATH }\""
        }]
      }
    ],
    // Bloker farlige PowerShell-kommandoer
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "powershell -File C:\\AL\\MyProject\\.claude\\hooks\\check-dangerous.ps1"
        }]
      }
    ],
    // Log token-forbrug efter hver session
    "Stop": [{
      "hooks": [{
        "type": "command",
        "command": "claude /cost --json >> $env:USERPROFILE\\.claude\\spend.log"
      }]
    }]
  }
}

Eksempel: PreToolUse hook der blokerer farlige operationer

C:\AL\MyProject\.claude\hooks\check-dangerous.ps1
# Læs tool input fra stdin (JSON)
$input = $input | ConvertFrom-Json
$cmd = $input.tool_input.command

# Bloker farlige patterns
$dangerous = @(
  'Remove-Item.*-Recurse.*-Force',
  'rm -rf',
  'DROP TABLE',
  'DROP DATABASE',
  'git push --force',
  'Uninstall-NAVApp'      # BC-specifik: uninstall apps
)

foreach ($pattern in $dangerous) {
  if ($cmd -match $pattern) {
    Write-Host "BLOKERET: Kommandoen indeholder farligt pattern: $pattern"
    exit 2  # Exit code 2 = bloker tool-kald
  }
}
exit 0  # Exit code 0 = tillåd

Pas på med hooks
Hooks kører som dig — med dine fulde rettigheder. Test scripts grundigt før du tilføjer dem. En dårligt skrevet PostToolUse hook der kalder al build på hvert eneste Edit kan gøre Claude Code uudholdeligt langsom.

Workflow 1: Auto-review på pull request

.github/workflows/claude-review.yml
name: Claude AL Code Review
on:
  pull_request:
    paths: ['**/*.al', '**/*.xlf']

jobs:
  review:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Hele historik for git diff

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with: { node-version: '20' }

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Get changed AL files
        id: changed
        run: |
          $files = git diff --name-only origin/${{ github.base_ref }}...HEAD --diff-filter=AM | Where-Object { $_ -match '\.al$' }
          $files | Out-File changed-files.txt
          Write-Output "count=$($files.Count)" >> $env:GITHUB_OUTPUT
        shell: pwsh

      - name: Run Claude review
        if: steps.changed.outputs.count != '0'
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          $prompt = "Lav code review på AL-filerne i changed-files.txt. Tjek for: PascalCase, ToolTip/Caption på UI, korrekt ErrorBehavior, og at standard BC-felter ikke modificeres direkte. Output som markdown med fil-overskrifter."
          claude --print --model sonnet $prompt > review.md
        shell: pwsh

      - name: Post review as PR comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const body = fs.readFileSync('review.md', 'utf8');
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: '## Claude AL Review\n\n' + body
            });

Token-optimering

Reference
⚡ Prompt Caching

Marker store statiske blokke (vendor list, AL schema) med cache_control: ephemeral. TTL: 5 min. Sparer op til 82% på gentagne kald.

⇅ Model Routing
